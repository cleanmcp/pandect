# Contributing to pandect

Thanks for considering a contribution. There are three things you can help with:

1. **Add a new stack** — propose a stack that isn't covered yet
2. **Update an existing stack** — flag drift, deprecation, or a new gotcha
3. **Improve the meta-prompt** — make every future rulebook better at once

## How rulebooks are generated

Every rulebook in `rulebooks/*.md` was generated by feeding [`RULEBOOK_GENERATOR.md`](https://github.com/cleanmcp/pandect/blob/main/rulebooks/) — the meta-prompt — into a fresh AI session (Claude Code, Codex, or Cursor with web search), with one input: the stack description.

The meta-prompt enforces:

- 21 mandatory sections, in a fixed order
- ≥20 ALWAYS rules, ≥15 NEVER rules, ≥20 blast-radius rows, ≥15 pitfalls per file
- Every version cited via live WebSearch on the day of generation (no training cutoffs)
- Zero banned phrases ("depending on", "you could", "consider using", "feel free to", etc.)
- Every config file inlined — no "see other doc"
- A First-PR Scaffold that produces a deployable hello-world if executed top-to-bottom

This consistency is the product. Don't bypass the meta-prompt unless you're improving the meta-prompt itself.

## Adding a new stack

1. Open a [New stack issue](https://github.com/cleanmcp/pandect/issues/new?template=new_stack.yml) describing:
   - The stack (e.g. "Solid + Supabase + Vercel")
   - Target OSes (macOS / Windows / Linux / iOS / Android / cross-platform)
   - Why this stack belongs alongside the existing 33
2. Wait for triage (we'll either invite you to PR or pick it up ourselves)
3. Generate the rulebook by running the meta-prompt against your stack
4. Open a PR adding `rulebooks/<short-name>.md`
5. Add the new short name + alias in:
   - `bin/pandect.mjs` `ALIASES` map (if the long name differs)
   - `README.md` stack table

The CI workflow will verify the file is present and the CLI lists it.

## Updating an existing stack

Stacks drift. A library deprecates, a major version drops, a security advisory lands. When that happens:

1. Open an [Update stack issue](https://github.com/cleanmcp/pandect/issues/new?template=update_stack.yml) describing what changed and why a regeneration is justified.
2. Either regenerate the full rulebook (preferred — keeps the file internally consistent) or, for a small fix, edit the relevant section directly.
3. PR the diff.

For the regeneration path, use the same meta-prompt as the original. Versions in the new rulebook should reflect WebSearch results on the regeneration date.

## Improving the meta-prompt

If you find a missing section type, an unhelpful banned phrase, or a quality bar that needs raising, edit `RULEBOOK_GENERATOR.md` and explain the change in the PR description. A meta-prompt change should typically be followed by regenerating at least 2–3 existing rulebooks to confirm the new prompt produces strictly better output.

## Code (CLI) changes

The CLI in `bin/pandect.mjs` is intentionally tiny — zero dependencies, ESM, runs on Node ≥ 20. Don't add a build step or TypeScript. Don't add runtime deps. If you need a flag, prefer adding it as ~20 lines of plain JS over reaching for a CLI framework.

CI runs the CLI on macOS / Windows / Linux × Node 20 / 22, so make sure it stays cross-platform (no `which`, no shell-only redirects in code paths that run on Windows).

## Release flow

Releases are automated:

1. Bump `package.json` `version` (semver — patch for fixes, minor for new stacks, major for CLI breaking changes)
2. Tag and push: `git tag v<x.y.z> && git push origin v<x.y.z>`
3. The `release.yml` workflow publishes to npm with provenance attestation and creates a GitHub Release with auto-generated notes

Maintainers should never run `npm publish` manually.

## Code of conduct

Be kind. Assume the other person is doing their best. We don't have a formal CoC document yet — if a situation arises where one is needed, we'll write one. In the meantime, please flag anything uncomfortable to the maintainers.
